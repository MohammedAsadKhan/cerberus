# Cerberus — Automated NIST Rules Update System

## Overview

Cerberus includes a GitHub Actions workflow that automatically monitors the official NIST SP 800-63B page for changes every week. When a change is detected in the sections relevant to password policy, the workflow opens a Pull Request with a full diff report and review checklist — so a human can assess the impact before any rules are updated.

This design is intentional. Compliance logic should never update silently. Every change to the ruleset goes through a review gate.

---

## How It Works

```
Every Monday at 08:00 UTC
        ↓
Workflow fetches NIST SP 800-63B (live)
        ↓
Extracts sections 5.1.1, 5.1.1.1, 5.1.1.2
        ↓
SHA-256 hashes the extracted content
        ↓
Compares against .github/nist_baseline_hash.txt
        ↓
No change → workflow exits cleanly, no action taken
Changed  → opens a Pull Request with full report
        ↓
Reviewer reads the diff, updates nist_rules.json if needed
        ↓
PR merged → users get updated rules on next --update-rules run
```

---

## Workflow File

```
.github/workflows/check_nist_update.yml
```

The workflow runs on two triggers:

| Trigger | When |
|---|---|
| Scheduled | Every Monday at 08:00 UTC |
| Manual | Anytime via the Actions tab → Run workflow |

---

## What Gets Monitored

The workflow targets the three sections of NIST SP 800-63B most directly relevant to password policy:

| Section | Title |
|---|---|
| 5.1.1 | Memorized Secret Authenticators |
| 5.1.1.1 | Memorized Secret Authenticator Requirements |
| 5.1.1.2 | Memorized Secret Verifier Requirements |

These sections govern minimum length, expiry policy, complexity rules, breach database checks, and related controls — everything Cerberus audits against.

---

## Pull Request Format

When a change is detected, the workflow opens a PR titled:

```
[NIST] SP 800-63B change detected — rules review required
```

The PR body includes:

- Date the change was detected
- Extracted text from each monitored section at the time of detection
- A review checklist
- Labels: `compliance`, `rules-update`, `needs-review`

The PR does **not** automatically modify `nist_rules.json`. That update is the responsibility of the reviewer.

---

## Reviewer Responsibilities

When a PR is opened by this workflow:

1. Read the extracted section content in the PR body
2. Compare it against the current `rules/nist_rules.json`
3. Determine whether any Cerberus rules need to change
4. If yes — update `rules/nist_rules.json`, increment `version`, update `last_updated`
5. Merge the PR

If the change is cosmetic (formatting, wording) and does not affect compliance logic, the PR can be merged as-is with no changes to `nist_rules.json`.

---

## Baseline Hash File

```
.github/nist_baseline_hash.txt
```

This file stores the SHA-256 hash of the last known NIST page content. It is updated automatically when the workflow detects a change and commits the new hash as part of the PR. Do not edit this file manually.

On first run, the baseline is set to `none`, which guarantees the workflow opens an initial PR to establish a baseline hash and confirm the monitoring is working correctly.

---

## Manual Trigger

To run the check outside of the weekly schedule:

1. Go to your repository on GitHub
2. Click the **Actions** tab
3. Select **NIST Rules Update Check**
4. Click **Run workflow** → **Run workflow**

This is useful after a known NIST publication event or to verify the workflow is functioning correctly.

---

## Required Permissions

The workflow uses `GITHUB_TOKEN` (automatically available in all GitHub Actions environments) to open Pull Requests. No additional secrets or configuration are required.
